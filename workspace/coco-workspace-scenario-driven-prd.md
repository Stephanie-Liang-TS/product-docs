## 一、产品定位

### 1.1 一句话定位

> **COCO Workspace 是一个双模式 AI 工作空间——用户在进入时选择"代码开发"或"日常办公"模式，系统自动适配界面、工具和 Agent 配置。人类和 AI Agent 作为平级团队成员，通过 HxA 架构协同完成工作。**

### 1.2 为什么是双模式？

v1.0 方案将用户拆分为三个细分场景（社媒/创意、游戏开发、开发项目），每个场景有独立的模块和界面。经过评审，该方案存在以下问题：

| 问题 | 说明 |
|------|------|
| 开发成本高 | 三套场景模板 = 三套界面 + 三套功能逻辑，工程量约为双模式的 1.5-2 倍 |
| 维护负担重 | 每个功能改动需要在三个场景下分别测试和适配 |
| 用户分类过于刚性 | 实际用户的需求往往跨越场景（如游戏开发者也需要社媒推广） |
| 市场验证不足 | 过度细分在 MVP 阶段是过度设计 |

参考 WorkBuddy（腾讯 CodeBuddy 团队产品）的设计思路——WorkBuddy 将功能分为"编程"和"办公"两大模式，用户在入口处选择，Workspace 自动适配。这一方案更简洁、更实用。

### 1.3 双模式定义

| 模式 | 名称 | 目标用户 | 涵盖场景 |
|------|------|---------|---------|
| **模式 A** | 代码开发 | 开发者、游戏开发者、技术用户 | 游戏开发(the9Bit)、开发项目(杜平)、代码协作、CI/CD、技术文档 |
| **模式 B** | 日常办公 | 非技术用户、运营、创作者 | 社媒运营(YOYO)、内容创作、文档处理、数据分析、研究调研、翻译、文件管理 |

### 1.4 与现有产品的关系

| 维度 | 现有 Dashboard | 新 Workspace |
|------|---------------|-------------|
| 定位 | 个人 AI 员工管理面板 | 团队 AI-Native 协作平台 |
| Agent 模型 | 1 用户 : 1 Agent (独立 VM) | 1 团队 : N Agent（多专业角色） |
| 交互 | 通过外部 IM（Lark/TG） | 内置对话界面 + 外部 IM 可选 |
| 任务 | 无 | 原生看板 + 模式自适应视图 |
| 代码库 | coco-dashboard (Next.js + Fastify) | 全新代码库 |
| 用户群 | 现有 300+ 个人用户 | 新产品线，面向团队和个人 |

> **关键决策：** 全新产品线，不是在现有 Dashboard 上改。现有 Dashboard 继续服务个人用户，Workspace 作为独立产品。两个产品共享 Logto 账户体系，代码库、部署、计费完全独立。

### 1.5 核心差异化

| 竞品 | 定位 | COCO 差异 |
|------|------|---------|
| Cursor / Windsurf | AI IDE（代码编辑器） | COCO 不是编辑器，是对话式工作空间，且覆盖非代码场景 |
| Devin | AI 程序员 | COCO 不只服务程序员，日常办公模式服务所有用户 |
| WorkBuddy | 桌面办公自动化 Agent | COCO 是 Web 平台 + HxA 团队协作，非桌面软件 |
| Canva | 设计工具 | COCO 有 AI Agent 能力，能自主执行复杂任务链 |
| Buffer / Later | 社媒排期 | COCO 能 AI 生成内容 + 排期 + 发布一站式 |
| Notion AI | AI 增强文档 | COCO 以对话为核心，Agent 是团队成员而非工具 |

### 1.6 目标用户与真实客户

| 客户 | 模式 | 场景 |
|------|------|------|
| **YOYO** | 日常办公 | KOL，使用 COCO 进行图片视频制作并自动更新社媒 |
| **the9Bit** | 代码开发 | 游戏开发项目，使用 COCO 进行项目管理和代码协作 |
| **杜平** | 代码开发 | 有实际开发项目在 COCO 上运行 |

### 1.7 成功标准

| 指标 | 上线 3 个月目标 | 上线 6 个月目标 |
|------|-------------|-------------|
| Workspace 月活用户 | 100（现有用户迁移） | 300 |
| MRR 增量 | +$10,000 | +$30,000 |
| 现有用户流失率 | < 10% | < 5% |
| NPS | > 30 | > 50 |
| 模式选择覆盖率 | 80%+ 新用户选择模式 | 90%+ |

## 二、双模式设计理念

### 2.1 模式切换机制

参照 WorkBuddy 的 toggle 设计，用户在 Workspace 入口选择模式，系统自适应调整：

```
┌─────────────────────────────────────────────────┐
│                                                 │
│  欢迎使用 COCO Workspace！                       │
│                                                 │
│  选择你的工作模式：                               │
│                                                 │
│  ┌────────────────────┐  ┌────────────────────┐ │
│  │  💻 代码开发         │  │  📋 日常办公         │ │
│  │                    │  │                    │ │
│  │  游戏开发、代码项目  │  │  社媒运营、内容创作  │ │
│  │  部署调试、Git 协作  │  │  文档处理、数据分析  │ │
│  │  CI/CD、API 开发    │  │  研究调研、文件管理  │ │
│  └────────────────────┘  └────────────────────┘ │
│                                                 │
│  模式可以随时在设置中切换                         │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 2.2 模式切换规则

| 规则 | 说明 |
|------|------|
| 首次进入时选择 | 新用户注册后首次进入 Workspace 时选择模式 |
| 随时可切换 | 在左侧导航栏顶部或设置中一键切换模式 |
| 数据不丢失 | 切换模式不影响已有数据，只改变界面布局和默认工具 |
| 记住选择 | 系统记住用户上次使用的模式，下次自动进入 |
| 团队级默认 | Workspace 管理员可设置团队默认模式 |

### 2.3 模式影响范围

模式切换影响的是**界面呈现和默认配置**，底层功能模块是共享的：

| 维度 | 代码开发模式 | 日常办公模式 |
|------|------------|------------|
| 左侧导航 | 对话 / 项目 / 任务 / 代码协作 / Agent / 工具箱 / 设置 | 对话 / 创作台 / 排期 / 任务 / 素材库 / Agent / 工具箱 / 设置 |
| 主页仪表板 | 项目进度、最近提交、部署状态、活跃 Agent | 待发内容、素材库快照、数据概览、快速操作 |
| 默认 Agent | 代码助手、测试 Agent、DevOps Agent | 文案助手、设计助手、排期助手 |
| 默认工具 | GitHub、GitLab、Vercel、Docker | 小红书、Instagram、DALL-E、排期工具 |
| 术语体系 | 技术术语（部署、仓库、API、容器） | 日常用语（发布、文件夹、连接、助手） |
| 任务标签 | 代码/测试/DevOps/文档/前端/后端 | 设计/文案/排期/数据/翻译/研究 |
| 快捷操作 | 写代码、部署、查日志、创建 PR | 生成配图、写文案、排期发布、数据分析 |

### 2.4 跨模式功能矩阵

以下功能在两种模式下的可见性和呈现方式：

| 功能模块 | 代码开发模式 | 日常办公模式 | 共享基础设施 |

## 三、核心设计原则

### 3.1 对话优先

**原则说明**：Workspace 中的一切操作都可以通过对话完成。视觉界面是对话的补充，不是替代。

**设计要求**：
- 每个模块都有对话入口：用户可以在对话中说"帮我生成一张春季穿搭的配图"，而不需要先导航到创作工作台
- 对话结果可以"钉"到视觉工作区：AI 生成的图片可以一键添加到素材库、排期日历或项目看板
- 支持上下文切换：在对话中讨论代码时，Agent 知道当前项目的上下文
- 不强制用户离开对话：如果用户只想在对话中完成所有事情，可以完全不使用视觉模块

**正面案例**：
```
用户: 帮我生成一张小红书封面，主题是"夏日防晒指南"，风格清新
Agent: 好的，我为你生成了 3 个方案：
       [图片1] [图片2] [图片3]
       
       你可以：
       → 选择一张直接发布
       → 修改任一张（"第2张换个背景色"）
       → 保存到素材库稍后使用
       
用户: 第2张不错，但文字再大一点
Agent: 已调整。[修改后的图片]
       → [加入今天的排期] [保存到素材库] [直接发布]
```

### 3.2 模式自适应（不是空白编辑器）

**原则说明**：新用户首次进入 Workspace 时，选择模式后立即看到适配好的界面——不需要从空白页面开始配置。

**设计要求**：
- 两种模式各有预配置的导航栏、主页、默认 Agent、默认工具
- 模式可以随时切换，切换即刻生效
- 每种模式内可以自定义（添加/移除模块、调整布局）
- 高级用户可以同时开启两种模式的功能（如开发者也需要社媒工具）

### 3.3 低门槛

**原则说明**：日常办公模式的主要用户是非技术人员。界面和交互必须对这些用户友好。

**设计要求**：
- 日常办公模式不出现技术术语
- 所有功能都可以通过自然语言对话完成
- 视觉界面用大图标和清晰标签
- 提供引导式操作：第一次使用每个功能时，Agent 主动引导
- 代码开发模式可以使用技术术语——目标用户本身就是技术人员

### 3.4 渐进式复杂度

**原则说明**：简单任务简单做，复杂任务有深度工具。四个层次：

| 层次 | 说明 | 示例（图片生成） |
|------|------|----------------|
| Level 1 — 对话 | 自然语言描述即可 | "帮我生成一张春天的封面图" |
| Level 2 — 快捷按钮 | 界面上的一键操作 | 点击"生成配图"按钮，填写简单表单 |
| Level 3 — 参数面板 | 展开高级参数 | 模型选择、尺寸、风格强度、Seed |
| Level 4 — 自动化 | 设置规则自动执行 | "每周一早9点自动生成本周排期内容的配图" |

### 3.5 HxA 协作理念

**原则说明**：Agent 不是"工具"，是"团队成员"。这是与所有竞品的核心差异——详见第四章。

### 3.6 上下文持久性

**原则说明**：Agent 记住用户偏好、历史操作和项目上下文。每次交互不从零开始。

**设计要求**：
- 日常办公模式：Agent 记住品牌风格偏好（配色、字体、调性）、常用操作模式
- 代码开发模式：Agent 记住项目上下文（代码库结构、当前 Sprint、最近的 Bug）
- 跨会话记忆：上周的对话内容在本周仍然可以引用

## 四、HxA 核心架构

> 本章保留自原始 General PRD 的 HxA (Human x Agent) 架构，这是 COCO Workspace 的核心突破点。

### 4.1 核心概念：Agent 是团队成员

传统协作工具里，AI 是"工具"——人调用它，用完丢掉。COCO Workspace 的核心突破是：**Agent 是团队成员**，有角色、有任务、有绩效、有权限。

```
Workspace（工作空间）
├── Team（团队）
│   ├── 人类成员 ─── 角色（管理员/成员/审核者）
│   └── Agent 成员 ─── 角色（执行者/审核者/协调者/观察者）
├── Mode（模式）
│   ├── 代码开发 ─── 开发相关的界面和工具
│   └── 日常办公 ─── 非代码相关的界面和工具
├── Projects（项目）
│   ├── Sprint / Phase 周期
│   └── Tasks（任务） ─── 分配给 人类 或 Agent
├── Conversations（对话）
│   ├── 频道对话（团队可见）
│   └── 私人对话（1:1）
├── Toolbox（工具箱）
│   ├── MCP 集成（GitHub/Lark/Jira/...）
│   ├── 内置技能（数据库查询/安全扫描/...）
│   └── 社区技能
└── Admin（管理）
    ├── 席位管理（人 + Agent 统一视图）
    ├── 权限控制（Agent 执行权限 + 审批策略）
    └── 用量分析（API 调用/Credits 消耗）
```

### 4.2 Agent 角色体系

| 角色 | 权限 | 代码开发模式典型场景 | 日常办公模式典型场景 |
|------|------|-------------------|-------------------|
| 执行者 | 接收任务 → 自主执行 → 提交结果 | 代码助手写代码、测试 Agent 跑测试 | 设计助手生成配图、文案助手写文案 |
| 审核者 | Review 他人产出 → 提出修改建议 | Code Review Agent | 内容审核 Agent（检查品牌合规） |
| 协调者 | 拆解需求 → 分配子任务 → 跟踪进度 | 项目管理 Agent | 排期管理 Agent |
| 观察者 | 只读监控 → 收集数据 → 生成报告 | 数据监控 Agent | 社媒数据分析 Agent |

### 4.3 统一消息总线 (UMB)

所有人类和 Agent 之间的通信走统一消息总线（Unified Message Bus），确保：

```
┌──────────────┐    ┌───────────────────┐    ┌──────────────┐
│  人类成员     │    │  Unified Message  │    │  Agent 成员   │
│              │───→│  Bus (UMB)        │←───│              │
│  Web UI      │    │                   │    │  代码助手     │
│  Telegram    │    │  - 消息路由       │    │  设计助手     │
│  Lark        │    │  - 权限检查       │    │  文案助手     │
│  WeChat      │    │  - 审计日志       │    │  测试 Agent   │
│              │    │  - 实时推送       │    │              │
└──────────────┘    └───────────────────┘    └──────────────┘
```

**UMB 核心能力**：
- **消息路由**：根据 @提及、频道归属、任务关联自动路由消息
- **权限检查**：Agent 执行操作前检查权限等级
- **审计日志**：所有操作可追溯
- **实时推送**：WebSocket 实时推送到所有客户端
- **跨渠道同步**：Workspace、Telegram、Lark 消息双向同步

### 4.4 Agent 执行权限模型

```
Agent 权限 = 角色基础权限 ∩ 管理员自定义限制

执行者 Agent 默认权限：
  ✓ 代码执行（在沙箱内）
  ✓ 文件读写（项目目录内）
  ✓ 外部 API 调用（白名单内）
  ✗ 数据库直接操作（需审批）
  ✗ 生产环境部署（需人工确认）
  ✗ 费用超限操作（需审批）

管理员可自定义：
  - 开关每项权限
  - 设置金额阈值
  - 指定审批人
  - 配置自动回滚策略
```

### 4.5 HxA 协作拓扑

可视化展示团队中人类与 Agent 的协作关系：

```
┌──────────────────────────────────────────────────┐
│  HxA 协作拓扑 — 产品开发团队                       │
│                                                  │
│      [Stephanie]          [Kevin]                │
│      产品经理              全栈开发                │
│         │                    │                   │
│         ├───── 指导 ────→ [代码助手 Agent]        │
│         │                    │                   │
│         ├───── 指导 ────→ [设计助手 Agent]        │
│         │                                        │
│      [Lisa]               [测试 Agent]           │
│      UI 设计师              ↑                     │
│         │                   │                    │
│         └─── 提交设计 → 审核 ┘                    │
```

### 6.1 对话 (Chat) — 通用核心

#### 6.1.1 模块概述

| 属性 | 值 |
|------|---|
| 模块名 | 对话 (Chat) |
| 优先级 | P0（核心模块，两种模式必备） |
| 目标用户 | 全部用户 |
| 核心定位 | 整个 Workspace 的通用入口和主要交互方式 |

Chat 模块是 COCO Workspace 的心脏。它不是附属功能，而是产品的核心入口。所有操作都可以通过对话完成，模式不同只影响 Agent 的默认行为和结果卡片的呈现方式。

#### 6.1.2 核心功能

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 对话主界面 | 与默认 Agent 的 1:1 对话。支持文本、图片、文件输入。流式响应。 | P0 |
| 富结果卡片 | Agent 返回的结果以卡片形式展示：图片预览卡、代码 diff 卡、排期日历卡、任务进度卡 | P0 |
| 操作按钮 | 结果卡片上附带操作按钮：保存/发布/修改/放弃/跳转到模块 | P0 |
| 消息归属 | 每条消息标记发送者（人类头像 or Agent 图标+名称） | P0 |
| 频道对话 | 团队公开频道，人和 Agent 都能参与。消息实时推送。 | P0 |
| 私人对话 | 1:1 跟 Agent 或跟人。Agent 私聊内容不对其他成员可见。 | P0 |
| @提及 | @人 或 @Agent 触发通知/唤醒 | P0 |
| 富文本 + 代码块 | Markdown 渲染、代码高亮、文件附件、图片 | P0 |
| 历史对话 | 按日期/项目/主题分组的对话历史，支持搜索 | P0 |
| 上下文切换 | 在对话中切换当前上下文（项目/频道/Agent） | P1 |
| 多 Agent 对话 | 在同一对话中 @不同的 Agent（@设计助手 帮我做个图） | P1 |
| 对话模板 | 根据模式不同显示不同的快捷入口 | P1 |
| 消息引用/回复 | 引用上下文回复，线程式讨论 | P1 |
| 消息搜索 | 全文搜索历史消息 | P1 |
| 语音输入 | 支持语音转文字（移动端尤其重要） | P1 |
| IM 桥接 | 与 Telegram/Lark/WeChat 双向同步 | P2 |

#### 6.1.4 对话意图识别与模式路由

Chat 模块内置意图识别，根据当前模式将用户意图路由到对应功能：

| 意图 | 日常办公模式路由 | 代码开发模式路由 |
|------|----------------|----------------|
| "帮我做一张图" | 创作工作台 → 图片生成卡片 | 创作工作台（若已启用） |
| "安排明天发布" | 排期日历 → 排期卡片 | — |
| "项目进度怎么样" | 任务列表 → 进度摘要 | 项目空间 → Sprint 进度卡片 |
| "帮我改代码" | — | 代码协作 → diff 卡片 |
| "帮我部署" | — | 代码协作 → 部署状态卡片 |
| "上周数据怎么样" | 数据分析 → 图表卡片 | — |
| "找上次的文件" | 素材库 → 搜索结果 | 文件管理 → 搜索结果 |

#### 6.1.5 技术要点

- 实时通讯：Socket.IO + Redis Pub/Sub 扇出
- 消息存储：PostgreSQL + 全文搜索（pg_trgm）
- Agent 响应：消息发送后通过 UMB 分发到对应 Agent 运行时，Agent 通过 SSE 回传流式响应
- 文件上传：GCS，消息中存引用
- 意图识别：基于 Agent 的自然语言理解能力，不需要独立的 NLU 模型

#### 6.4.5 Agent 运行时架构

> **关键架构决策：** 复用现有 Zylos Runtime，不重建 Agent 运行时。

| 方案 | 说明 | 推荐 |
|------|------|------|
| 方案 A：复用 Zylos Runtime | Workspace 通过 API 调用现有 Zylos Agent | **强烈推荐** |
| 方案 B：Docker 容器 | 每个 Agent 独立容器 | 备选（成本高） |
| 方案 C：独立 VM | 每个 Agent 独立 VM | 不推荐（成本极高） |

**推荐**：Workspace 不重新构建 Agent 运行时。通过 API 接口调用现有 Zylos 运行时。Workspace 是前端 + API 层，Agent 智能由 Zylos 提供。这大幅降低开发成本和复杂度。

## 八、技术选型建议

### 8.1 技术栈总览

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 前端框架 | Next.js 16 (App Router) | SSR + RSC，团队已有经验 |
| UI 框架 | Tailwind CSS v4 + shadcn/ui | 设计稿风格匹配，组件生态成熟 |
| 状态管理 | Zustand | 轻量，适合中等复杂度 |
| 实时通讯 | Socket.IO + Redis Adapter | 对话模块 WebSocket + UMB 消息分发 |
| 后端框架 | Fastify (TypeScript) | 团队已有经验，高性能 |
| ORM | Prisma 6 | Type-safe，团队已有经验 |
| 数据库 | PostgreSQL 17 | 复用 Cloud SQL |
| 缓存/实时 | Redis 7 (Pub/Sub + Cache) | 消息扇出、会话缓存、在线状态 |
| 任务队列 | BullMQ | Agent 任务调度、图片生成队列、异步 Job |
| Agent 运行时 | 复用 Zylos Runtime（API 调用） | 不重建运行时，大幅降低成本 |
| 文件存储 | GCS (Google Cloud Storage) | 文件附件、AI 产出物 |
| CDN | Cloudflare | 静态资源和图片分发 |
| 认证 | Logto（共享现有实例） | SSO + 社交登录 + RBAC |
| 图片处理 | Sharp (服务端) + Fabric.js (客户端) | 图片编辑器 MVP 方案 |
| 图片生成 | DALL-E 3 API（默认）+ Stability AI（备选） | 多模型支持 |
| 搜索 | PostgreSQL pg_trgm + GiST | 全文搜索（初期），后期可迁 Meilisearch |

## 十、风险与决策点

### 10.1 核心风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| **双模式边界模糊** | 中——用户不确定选哪个模式 | 中 | 提供清晰的模式说明 + 随时可切换 + 高级用户可同时开启两模式功能 |
| **AI 图片生成质量不稳定** | 中——影响日常办公用户体验 | 中 | 多模型支持(DALL-E + SD + Flux)，用户可切换；质量问题走"重新生成" |
| **社媒平台 API 限制** | 中——影响自动发布 | 高 | MVP 做排期 + 通知提醒，不做真实自动发布。Phase 2 接入 API |
| **图片编辑器开发复杂度** | 中——可能延期 | 中 | MVP 只做基础编辑 + AI 编辑，不做专业 PS 级功能 |
| **Agent 运行时可靠性** | 中——Agent 执行可能失败 | 中 | 超时机制、重试策略、人工兜底通知 |
| **IM-Workspace 同步复杂度** | 低——不影响 MVP | 高 | MVP 不做同步，Phase 2 再做 |
| **用户对新产品的接受度** | 中——现有用户可能不愿切换 | 中 | 不强制迁移。Workspace 作为增值功能 |

### 10.2 关键决策点

#### 决策 1：模式边界如何处理交叉需求？

| 选项 | 说明 |
|------|------|
| A: 严格隔离 | 日常办公用户完全看不到代码功能 |
| B: 默认隔离 + 可手动开启 | 用户在设置中可以开启另一模式的功能 |
| C: 全部可见 + 模式只影响排序 | 所有功能都可见，模式只影响导航排序和推荐 |

**建议：选项 B**。默认按模式隔离降低认知负担，但允许高级用户跨模式使用功能。例如游戏开发者（代码开发模式）也需要生成美术素材，可以手动开启创作工作台。

#### 决策 2：MVP 两个模式是否同时开发？

| 选项 | 说明 |
|------|------|
| A: 同时开发两个模式 | 覆盖面广，但工期可能超 12 周 |
| B: 先做日常办公，后做代码开发 | 先服务最大用户群(55-65%) |
| C: 先做代码开发，后做日常办公 | 先服务高付费用户 |

**建议：选项 A**。双模式的核心价值在于"切换"体验——如果 MVP 只有一个模式，则与 v1.0 无本质区别。两个模式共享大量基础设施（Chat/Agent/管理后台），增量工作主要在模式特定视图上。

#### 决策 3：图片生成用哪个模型？

| 模型 | 质量 | 速度 | 成本 |
|------|------|------|------|
| DALL-E 3 | 高 | 中(5-15s) | $0.04-$0.08/张 |
| Stability AI | 中高 | 快(2-5s) | $0.01-$0.03/张 |
| Flux | 高 | 中 | 变化中 |

**建议：MVP 用 DALL-E 3 默认 + Stability AI 备选**。Flux 作为 P1。

#### 决策 4：排期发布是否接入真实平台 API？

**建议：MVP 不接入**。做排期日历 + Telegram/Lark 通知提醒（"你排期的内容该发布了，点击复制文案和图片"）。Phase 2 接入平台 API。

#### 决策 5：代码协作的深度？

| 选项 | 说明 | 成本 |
|------|------|------|
| A: 纯对话式 | 所有代码操作在对话中完成 | 低 |
| B: 对话 + 基础代码查看器 | 对话式操作 + 文件/代码浏览 | 中 |
| C: 对话 + 完整编辑器 | 类 Monaco Editor | 高 |

**建议：MVP 用选项 A，Phase 2 做选项 B**。不做在线编辑器——那会走回 IDE 路线。
