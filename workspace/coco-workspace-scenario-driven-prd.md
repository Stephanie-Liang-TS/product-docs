## 一、产品定位

### 1.1 一句话定位

> **COCO Workspace 是一个场景驱动的AI工作空间——以对话为核心交互方式，根据用户场景动态呈现工具和界面，让每个人都能通过AI Agent高效完成自己的工作。**

### 1.2 产品定位说明

COCO Workspace **不是一个IDE**。它不是VS Code、不是Cursor、不是Devin。

COCO Workspace **是一个智能助手控制台**（Smart Assistant Dashboard）——用户通过对话告诉AI自己要做什么，workspace根据场景呈现合适的工具和界面。社媒运营者看到的是创作工作台和排期日历；游戏开发者看到的是项目看板和资产生成器；开发者看到的是对话式代码协作空间。

### 1.3 为什么不做IDE？

| 维度 | 数据支撑 |
|------|---------|
| 用户构成 | 88%是个人用户，无公司名，非开发团队 |
| 使用场景 | CRM中零用户有开发/编程/代码相关场景记录 |
| 交互方式 | 100%用户通过Telegram/Lark对话使用 |
| 最大用户群 | 55-65%是社媒/创意用户，从不写代码 |
| 竞争格局 | IDE赛道：Cursor估值$9B、Devin估值$2B。IM Agent赛道：几乎无直接竞品 |
| 财务风险 | IDE转型将疏远80%+现有付费用户，威胁$33K+/月MRR |

### 1.4 核心差异化

| 竞品 | 定位 | COCO差异 |
|------|------|---------|
| Cursor / Windsurf | AI IDE（代码编辑器） | COCO不是编辑器，是对话式工作空间 |
| Devin | AI程序员 | COCO不只服务程序员，服务所有用户 |
| Canva | 设计工具 | COCO有AI Agent能力，能自主执行复杂任务 |
| Buffer / Later | 社媒排期 | COCO能AI生成内容+排期+发布一站式 |
| Notion AI | AI增强的文档工具 | COCO以对话为核心，不是文档编辑器 |
| Midjourney | AI生图 | COCO将生图与项目管理、发布集成 |

### 1.5 目标用户

本产品面向COCO现有的三大用户群体和潜在的新用户：

| 群体 | 占比 | 技术水平 | 核心诉求 |
|------|------|---------|---------|
| 社媒/创意用户 | 55-65% | 低 | "帮我做图、写文案、排期发布" |
| 游戏开发者 | 15-20% | 中 | "帮我管理项目、生成素材、写代码" |
| 开发项目用户 | 15-20% | 高 | "帮我写代码、部署、调试，但别让我用IDE" |

### 1.6 成功标准

| 指标 | 上线3个月目标 | 上线6个月目标 |
|------|-------------|-------------|
| Workspace 月活用户 | 100（现有用户迁移） | 300 |
| MRR增量 | +$10,000 | +$30,000 |
| 现有用户流失率 | < 10%（不因产品转型流失） | < 5% |
| NPS | > 30 | > 50 |
| 场景模板使用率 | 70%新用户选择场景模板 | 85% |
| Air→Pro转化率 | 10% | 20% |

## 三、核心设计原则

### 3.1 对话优先（不是IDE优先）

**原则说明**：workspace中的一切操作都可以通过对话完成。视觉界面是对话的补充，不是替代。

**设计要求**：
- 每个模块都有对话入口：用户可以在对话中说"帮我生成一张春季穿搭的配图"，而不需要先导航到创作工作台
- 对话结果可以"钉"到视觉工作区：AI生成的图片可以一键添加到素材库、排期日历或项目看板
- 支持上下文切换：在对话中讨论代码时，Agent知道当前项目的上下文；讨论配图时，Agent知道品牌风格偏好
- 不强制用户离开对话：如果用户只想在对话中完成所有事情，可以完全不使用视觉模块

**反面案例（避免）**：
- 不要做成"必须先在界面上选择项目 → 打开代码编辑器 → 才能让Agent改代码"
- 不要让用户必须通过表单填写来创建内容——对话就是表单

**正面案例**：
```
用户：帮我生成一张小红书封面，主题是"夏日防晒指南"，风格清新、色彩明亮
Agent：好的，我为你生成了3个方案，你看看哪个合适？
      [图片1] [图片2] [图片3]
      
      你可以：
      → 选择一张直接发布
      → 修改任一张（"第2张换个背景色"）
      → 保存到素材库稍后使用
      
用户：第2张不错，但文字再大一点，加上日期"6月15日"
Agent：好的，已调整。[修改后的图片]
      → [加入今天的排期] [保存到素材库] [直接发布]
```

### 3.3 低门槛（非技术用户也能用）

**原则说明**：55-65%的用户是非技术用户。Workspace的主要交互方式必须对这些用户友好。

**设计要求**：
- 不出现任何技术术语（除非用户选择了开发者模板）
  - 不说"API"，说"连接"
  - 不说"部署"，说"发布"
  - 不说"Git仓库"，说"代码项目"
- 所有功能都可以通过自然语言对话完成
- 视觉界面用大图标和清晰标签，不用密集的文字列表
- 提供引导式操作：第一次使用每个功能时，Agent主动引导
- 错误信息用日常语言，不显示技术错误码

**术语映射表**：

| 技术术语 | 社媒/创意用户看到的 | 游戏开发者看到的 | 开发者看到的 |
|----------|-------------------|-----------------|-------------|
| Deploy | 发布 | 发布到平台 | 部署 |
| Repository | — | 项目文件夹 | 代码仓库 |
| API | 连接服务 | 连接服务 | API |
| CI/CD | — | 自动构建 | CI/CD |
| Container | — | — | 容器 |
| Agent Instance | 我的助手 | 我的AI队友 | Agent实例 |
| Webhook | 自动通知 | 自动通知 | Webhook |
| Token | 连接密钥 | 连接密钥 | Token |
| Pipeline | 工作流程 | 构建流程 | Pipeline |
| Staging | 预览版 | 测试版 | Staging |

### 3.4 渐进式复杂度（简单任务简单做，复杂任务有深度工具）

**原则说明**：简单任务不应该需要复杂的操作。但当用户需要更精细的控制时，深度工具应该可用。

**设计要求**：
- **Level 1 — 对话完成**：所有基础操作都可以通过对话完成（"帮我生成一张配图"）
- **Level 2 — 快捷操作**：常用操作在界面上有一键按钮（"生成配图"按钮）
- **Level 3 — 参数控制**：高级用户可以展开详细参数面板（图片尺寸、风格参数、Seed等）
- **Level 4 — 自动化**：资深用户可以设置自动化规则（"每周一早9点自动发布本周排期内容"）

**示例：图片生成的四层复杂度**

```
Level 1 (对话):
  用户: "帮我生成一张春天的封面图"
  Agent: [生成图片] 好的，这是生成的封面图

Level 2 (快捷按钮):
  用户点击 [🖼 生成配图] → 弹出简单表单：
  - 主题: ___
  - 风格: [清新] [商务] [科技] [手绘]
  - 平台: [小红书] [Instagram] [通用]
  → [生成]

Level 3 (参数面板):
  展开"高级选项" ↓
  - 模型: [DALL-E 3] [Midjourney v6] [Stable Diffusion]
  - 宽高比: [1:1] [4:3] [9:16] [16:9] [自定义]
  - 风格强度: [轻] ■■■□□ [重]
  - 负面提示词: ___
  - 随机种子: ___
  - 生成数量: [1] [2] [4]

Level 4 (自动化):
  设置自动化规则：
  - 触发条件: 每周一早8:00
  - 动作: 基于本周排期的主题，自动生成3组配图方案
  - 输出: 保存到素材库"待审核"文件夹
  - 通知: 通过Telegram发送预览
```

### 3.5 上下文持久性

**原则说明**：Agent应该记住用户的偏好、历史操作和项目上下文。每次交互不应该从零开始。

**设计要求**：
- Agent记住用户的品牌风格偏好（配色、字体、调性）
- Agent记住常用的操作模式（"这个用户每次生成小红书配图都用竖版9:16"）
- 项目上下文在对话中自动关联（在讨论某个项目时，Agent知道项目的代码库、当前Sprint、最近的Bug）
- 跨会话记忆：上周的对话内容在本周仍然可以引用

### 3.6 即时反馈与可视预览

**原则说明**：任何AI生成的结果都应该有即时的可视化预览，用户可以立即看到效果并修改。

**设计要求**：
- AI生成的图片直接在对话中预览（不需要打开新页面）
- AI修改的代码在对话中用diff视图展示（删除/新增行高亮）
- AI生成的排期在对话中用小型日历视图预览
- 所有预览都支持一键操作（保存/发布/修改/放弃）

## 五、模块设计

### 5.1 Chat/对话 — 通用入口

#### 5.1.1 模块概述

| 属性 | 值 |
|------|---|
| 模块名 | Chat/对话 |
| 优先级 | P0（核心模块，所有场景必备） |
| 目标用户 | 全部用户 |
| 核心定位 | 整个Workspace的通用入口和主要交互方式 |

Chat模块是COCO Workspace的心脏。它不是附属功能，而是整个产品的核心入口。用户通过对话可以完成workspace中的所有操作。

#### 5.1.2 与CODE-YI Chat模块的差异

| 维度 | CODE-YI Chat | 场景驱动Chat |
|------|-------------|-------------|
| 定位 | 团队内部沟通工具 | 用户与AI Agent的主要交互界面 |
| 设计参考 | 飞书/Slack | 更像ChatGPT/Claude，但带有操作能力 |
| 对话对象 | 人与人、人与Agent、频道 | 主要是用户与Agent（1:1为主） |
| 核心功能 | 消息收发、@提及、频道 | 对话完成任务、结果预览、操作按钮 |
| 技术重心 | WebSocket实时通讯 | AI响应流式展示 + 富结果卡片 |

#### 5.1.3 核心功能

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 对话主界面 | 与默认Agent的1:1对话。支持文本、图片、文件输入。流式响应。 | P0 |
| 富结果卡片 | Agent返回的结果以卡片形式展示：图片预览卡、代码diff卡、排期日历卡、任务进度卡 | P0 |
| 操作按钮 | 结果卡片上附带操作按钮：保存/发布/修改/放弃/跳转到模块 | P0 |
| 历史对话 | 按日期/项目/主题分组的对话历史，支持搜索 | P0 |
| 上下文切换 | 在对话中切换当前上下文（项目/频道/Agent） | P1 |
| 多Agent对话 | 在同一个对话中@不同的Agent（@设计助手 帮我做个图） | P1 |
| 语音输入 | 支持语音转文字输入（移动端尤其重要） | P1 |
| 对话模板 | 常用对话起始模板："生成配图"、"写文案"、"创建项目" | P1 |
| IM桥接 | 对话内容与Telegram/Lark/WeChat同步（可选） | P2 |
| 协作对话 | 多人在同一对话中与Agent交互（团队场景） | P2 |

### 7.4 关键技术决策

#### 7.4.1 图片编辑器技术方案

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| Fabric.js (Canvas) | 成熟、功能完整、社区大 | 性能一般，移动端体验差 | **MVP推荐** |
| Konva.js (Canvas) | 性能好、React集成好 | 功能不如Fabric全面 | 备选 |
| PixiJS (WebGL) | 性能极好 | 学习曲线陡，生态小 | 不推荐 |
| 集成现有编辑器(Photopea API) | 功能极完整 | 依赖第三方、定制性差 | P2考虑 |

**推荐**：MVP阶段使用Fabric.js，提供基础编辑+AI编辑能力。AI编辑部分调用后端API（去背景、扩图等），前端只做预览和基础操作。

#### 7.4.2 实时通讯方案

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| Socket.IO | 成熟、自动重连、房间管理 | 非标准协议 | **推荐** |
| 原生WebSocket | 标准协议、轻量 | 需自己实现重连/心跳 | 备选 |
| SSE (Server-Sent Events) | 简单、HTTP兼容 | 仅服务端→客户端 | Agent响应流可用 |

**推荐**：Socket.IO用于双向实时通讯（对话、通知），SSE用于Agent响应的流式展示。

#### 7.4.3 Agent运行时方案

| 方案 | 说明 | 推荐 |
|------|------|------|
| 复用现有Zylos Runtime | Workspace通过API调用现有的Zylos Agent | **强烈推荐** |
| 新建Agent容器 | 每个Agent独立容器 | 不推荐（成本高、重复建设） |

**推荐**：Workspace不重新构建Agent运行时。通过API接口调用现有Zylos运行时。Workspace是前端+API层，Agent智能由Zylos提供。这大幅降低了开发成本和复杂度。

## 九、风险与决策点

### 9.1 核心风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| **管理层不同意放弃CODE-YI路线** | 高——方向性分歧导致项目无法推进 | 中 | 用本报告数据说明80%+用户不需要IDE，呈现具体收入风险数字 |
| **AI图片生成质量不稳定** | 中——影响创意用户体验 | 中 | 多模型支持(DALL-E + SD + MJ)，用户可切换；质量问题走"重新生成"流程 |
| **社媒平台API限制** | 中——影响自动发布功能 | 高 | MVP阶段先做手动发布+通知提醒，再逐步接入API；部分平台(小红书)可能无官方API，需评估替代方案 |
| **图片编辑器开发复杂度** | 中——可能延期 | 中 | MVP只做基础编辑+AI编辑，不做专业PS级功能；使用Fabric.js降低开发成本 |
| **Agent运行时性能** | 中——大量并发图片生成可能导致队列积压 | 低 | BullMQ队列管理+并发限制+优先级排序 |
| **IM-Workspace同步复杂度** | 低——不影响MVP | 高 | MVP阶段不做同步，Phase 2再做。先让两个渠道独立运行 |
| **用户对新产品的接受度** | 中——现有用户可能不愿意切换 | 中 | 不强制迁移。Workspace作为增值功能，IM继续正常使用 |

### 9.2 关键决策点

#### 决策1：是否完全放弃CODE-YI路线？

| 选项 | 利弊 |
|------|------|
| A: 完全放弃CODE-YI，转向场景驱动 | 利：资源集中、方向明确、服务现有用户。弊：放弃IDE市场机会 |
| B: 双轨并行（场景驱动为主、CODE-YI为辅） | 利：覆盖面广。弊：资源分散、团队撕裂 |
| C: 先做场景驱动MVP，根据数据再决定 | 利：用数据验证。弊：决策推迟 |

**建议：选项A**。原因：
1. 现有6-7人的团队无法支撑双轨并行
2. CODE-YI面向的开发者市场仅占15-20%用户
3. IDE赛道竞争极度激烈，COCO没有差异化优势
4. 场景驱动方案中已包含对开发者的支持（项目空间+对话式代码协作）

#### 决策2：MVP优先做哪个场景模板？

| 选项 | 覆盖群体 | 收入影响 | 开发难度 |
|------|---------|---------|---------|
| A: 社媒创作者（推荐） | 55-65%用户 | 保住$18K+/月 | 中等 |
| B: 游戏开发者 | 15-20%用户 | $11K-$15K/月 | 较高 |
| C: 开发者 | 15-20%用户 | $16K-$22K/月 | 最高 |

**建议：选项A**。社媒创作者是最大群体，且开发难度相对可控。Chat+AI图片生成+文案+排期是可以在8-10周内交付的MVP范围。

#### 决策3：图片生成用哪个模型？

| 模型 | 质量 | 速度 | 成本 | API可用性 |
|------|------|------|------|----------|
| DALL-E 3 | 高 | 中（5-15s） | $0.04-$0.08/张 | 稳定 |
| Stability AI (SDXL) | 中高 | 快（2-5s） | $0.01-$0.03/张 | 稳定 |
| Midjourney | 极高 | 慢（15-60s） | ~$0.01/张 | 非官方API，不稳定 |
| Flux | 高 | 中 | 变化中 | 稳定 |

**建议：MVP用DALL-E 3作为默认**，同时集成Stability AI作为快速/低成本选项。Midjourney作为P2功能（等官方API）。

#### 决策4：排期发布是否接入真实社媒平台API？

| 选项 | 利弊 |
|------|------|
| A: MVP就接入（至少Instagram） | 利：完整体验。弊：API集成复杂、审核时间长 |
| B: MVP只做排期管理，手动发布 | 利：快速上线。弊：体验不完整 |
| C: MVP做排期+通知提醒 | 利：接近完整体验、无API依赖。弊：不是真自动化 |

**建议：选项C**。排期日历+Telegram/Lark通知提醒（"你排期的小红书帖子该发布了，点击复制文案和图片"）。Phase 2再接入真实平台API。

#### 决策5：代码协作的深度？

| 选项 | 说明 | 开发成本 |
|------|------|---------|
| A: 纯对话式（无代码视图） | 所有代码操作在对话中完成，只有diff卡片 | 低 |
| B: 对话+基础代码查看器 | 对话式操作 + 可以浏览文件/查看代码 | 中 |
| C: 对话+完整代码编辑器 | 类Monaco Editor的在线编辑器 | 高 |

**建议：选项B**。在对话中完成核心操作（修改/提交/部署），提供只读的代码浏览器用于查看上下文。不做在线编辑器——那会走回IDE路线。

### 9.3 与CODE-YI方案的对比总结

| 维度 | CODE-YI (IDE路线) | 场景驱动Workspace |
|------|-------------------|-------------------|
| 目标用户 | 开发团队 | 所有现有用户 |
| 用户覆盖 | 15-20% | 90%+ |
| 交互方式 | IDE面板操作 | 对话优先 |
| 核心模块 | 代码编辑器、Git、任务看板 | Chat、创作工作台、项目空间 |
| 竞品 | Cursor ($9B)、Devin ($2B) | 无直接竞品 |
| 开发周期 | 24周（MVP到内测） | 10周（Phase 1 MVP） |
| 团队需求 | 6-7人 | 6-7人（相同） |
| 收入风险 | 高（疏远80%+用户） | 低（服务所有用户） |
| 市场定位 | 跟随（做别人做过的） | 引领（做别人没做的） |
| 技术复杂度 | 高（Agent容器化/IDE集成） | 中等（复用Zylos运行时） |

### 9.4 成功定义

**Phase 1 MVP成功标准（上线后1个月内）**：

| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| Workspace注册用户 | > 50 | 产品数据 |
| 日活用户 | > 20 | 日均活跃会话数 |
| AI图片生成使用量 | > 500张/月 | API调用统计 |
| 用户留存（7日） | > 40% | 登录数据 |
| NPS | > 20 | 用户调研 |
| 现有IM用户流失率 | < 5% | Stripe取消数据 |
| 关键错误 | < 3个/周 | 错误监控 |

**如果上述指标未达标**：
- 收集用户反馈，找出核心问题
- 优先修复体验问题，而非添加新功能
- 考虑是否需要调整场景模板的设计
- 不急于进入Phase 2，先确保Phase 1稳固
